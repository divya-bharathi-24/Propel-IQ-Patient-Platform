# Architecture Design

## Project Overview

The Unified Patient Access & Clinical Intelligence Platform is a standalone healthcare system that bridges patient scheduling with clinical data management. It serves three user roles (Patient, Staff, Admin) and combines intelligent appointment booking, AI-powered clinical document extraction, and Trust-First medical coding into a single platform. The system targets a reduction of no-show rates from a 15% baseline, eliminates manual clinical data extraction (20 minutes to 2 minutes), and achieves an AI-Human Agreement Rate exceeding 98%. Phase 1 operates exclusively on free/open-source hosting infrastructure.

## Architecture Goals

- AG-1: Trust-First AI Architecture — every AI-generated output flows through a mandatory human verification workflow before finalization, ensuring clinical safety and accountability
- AG-2: HIPAA Compliance by Design — encryption (AES-256 at rest, TLS 1.2+ in transit), immutable audit logging, and role-based access control are embedded at every architectural layer
- AG-3: Modular Monolith — single deployable unit with distinct bounded modules (Auth, Patient, Appointment, Clinical, AI, Notification, Admin) enabling team autonomy within a deployable footprint compatible with free-tier hosting; clear boundaries enable future extraction to microservices post-Phase 1
- AG-4: Free-Tier Deployability — all technology choices provide viable free-tier hosting options for Phase 1, with documented migration paths to paid infrastructure
- AG-5: Measurable Outcomes — architecture supports built-in metrics collection for AI agreement rate, no-show reduction, and clinical prep time benchmarks
- AG-6: Graceful Degradation — external service failures (calendar APIs, SMS/email, AI model providers) degrade functionality without blocking core booking and clinical workflows

## Non-Functional Requirements

### Performance

- NFR-001: System MUST respond to user-facing API requests within 2 seconds under normal load (p95 latency)
- NFR-002: System MUST respond to AI-powered operations (document extraction, medical coding) within 30 seconds (p95 latency), displaying progress indicators for operations exceeding 5 seconds
- NFR-012: System MUST render the frontend initial page load within 3 seconds on a standard broadband connection (LCP < 2.5s, FID < 100ms, CLS < 0.1)
- NFR-019: System MUST process uploaded PDF documents up to 50 MB without timeout or memory exhaustion
- NFR-020: System MUST display real-time appointment slot availability with no more than 5-second staleness

### Security

- NFR-004: System MUST encrypt all patient data at rest using AES-256 encryption or equivalent
- NFR-005: System MUST encrypt all data in transit using TLS 1.2 or higher
- NFR-006: System MUST enforce role-based access control at the API level for all endpoints, restricting access based on the authenticated user's role (Patient, Staff, Admin)
- NFR-007: System MUST automatically terminate user sessions after 15 minutes of inactivity and redirect to the login page
- NFR-008: System MUST store passwords using bcrypt or Argon2 hashing with minimum complexity enforcement (8+ characters, mixed case, number, special character)
- NFR-013: System MUST comply with HIPAA Privacy Rule and Security Rule requirements for all Protected Health Information (PHI) handling
- NFR-014: System MUST implement input validation and output encoding on all user-facing inputs to prevent injection attacks (OWASP A03)
- NFR-015: System MUST log all authentication events (login, logout, session timeout, failed attempts) with timestamps and source IP
- NFR-017: System MUST implement rate limiting on public-facing API endpoints to prevent abuse

### Availability

- NFR-003: System MUST maintain 99.9% uptime measured monthly, excluding planned maintenance windows
- NFR-018: System MUST provide graceful degradation when external services (calendar APIs, SMS/email providers, AI model providers) are unavailable, ensuring core booking and clinical workflows remain functional

### Scalability

- NFR-010: System MUST support concurrent access by a minimum of 100 simultaneous users without performance degradation
- NFR-016: System MUST support horizontal scaling of backend services without application code changes

### Reliability

- NFR-009: System MUST maintain an immutable audit log for all user actions involving patient data, recording user ID, timestamp, action type, and affected record
- NFR-011: System MUST achieve an AI-Human Agreement Rate exceeding 98% for suggested clinical data and medical codes

## Data Requirements

### Data Structures

- DR-001: System MUST store patient profiles with email as a unique identifier, containing name, phone number, date of birth, and hashed password
- DR-002: System MUST store appointment records with provider specialty, date, time slot, status (Booked, Arrived, Cancelled, Completed), patient reference, and cancellation reason
- DR-003: System MUST store waitlist entries linking a patient's active booking to a preferred unavailable slot, with FIFO ordering enforced via enrollment timestamp
- DR-004: System MUST store patient intake data (demographics, medical history, symptoms, medications) with a source indicator (AI conversational or manual form)
- DR-005: System MUST store uploaded clinical documents as encrypted binary objects with metadata (file name, upload date, processing status, patient reference)
- DR-006: System MUST store AI-extracted clinical data (vitals, medical history, medications, allergies, diagnoses) linked to their source document with per-field extraction confidence scores
- DR-007: System MUST store medical code suggestions (ICD-10, CPT) with AI confidence scores, source document references, and staff verification status (Pending, Accepted, Rejected, Modified)
- DR-008: System MUST store data conflict records with conflicting field values, source document references, and resolution status (Unresolved, Resolved, Pending Review)
- DR-014: System MUST store insurance validation records with provider name, insurance ID, validation result (Matched, Not Matched), and timestamp
- DR-015: System MUST store notification delivery records (SMS/Email) with delivery status, timestamp, channel, and retry count
- DR-016: System MUST store same-day queue entries with patient reference, queue position, arrival time, and status
- DR-017: System MUST store calendar sync records with external calendar provider, external event ID, sync status, and patient reference
- DR-018: System MUST store no-show risk scores per appointment with contributing factors and calculation timestamp

### Data Integrity

- DR-009: System MUST maintain referential integrity between patients, appointments, documents, intake records, and medical codes through foreign key constraints
- DR-010: System MUST implement soft deletion for patient records and appointment records, preserving data for audit and compliance purposes

### Data Retention

- DR-011: System MUST retain audit log records for a minimum of 7 years in compliance with HIPAA record retention requirements

### Data Backup

- DR-012: System MUST perform automated database backups at minimum daily frequency with point-in-time recovery capability within a 24-hour window

### Data Migration

- DR-013: System MUST support zero-downtime database schema migrations using versioned migration scripts managed by Entity Framework Core

### Domain Entities

- **Patient**: Represents a registered patient. Attributes: id (UUID), name, email (unique), phone, dateOfBirth, passwordHash, emailVerified (boolean), status (Active/Deactivated), createdAt. Relationships: one-to-many with Appointment, IntakeRecord, ClinicalDocument, InsuranceValidation, CalendarSync
- **User**: Represents an authenticated system user across all roles. Attributes: id (UUID), email (unique), passwordHash, role (Patient/Staff/Admin), status (Active/Deactivated), lastLoginAt, createdAt. Relationships: one-to-many with AuditLog
- **Appointment**: Represents a scheduled appointment slot. Attributes: id (UUID), patientId (FK), specialtyId (FK), date, timeSlotStart, timeSlotEnd, status (Booked/Arrived/Cancelled/Completed), cancellationReason, createdBy (FK), createdAt. Relationships: many-to-one with Patient, Specialty; one-to-many with Notification, NoShowRisk; one-to-one with QueueEntry
- **WaitlistEntry**: Represents a patient's preferred slot preference. Attributes: id (UUID), patientId (FK), currentAppointmentId (FK), preferredDate, preferredTimeSlot, enrolledAt, status (Active/Swapped/Expired). Relationships: many-to-one with Patient, Appointment
- **IntakeRecord**: Represents collected patient intake data. Attributes: id (UUID), patientId (FK), appointmentId (FK), source (AI/Manual), demographics (JSONB), medicalHistory (JSONB), symptoms (JSONB), medications (JSONB), completedAt. Relationships: many-to-one with Patient, Appointment
- **ClinicalDocument**: Represents an uploaded clinical PDF. Attributes: id (UUID), patientId (FK), fileName, fileSize, storagePath, mimeType, processingStatus (Pending/Processing/Completed/Failed), uploadedAt. Relationships: many-to-one with Patient; one-to-many with ExtractedData
- **ExtractedData**: Represents AI-extracted structured data from a clinical document. Attributes: id (UUID), documentId (FK), patientId (FK), dataType (Vital/History/Medication/Allergy/Diagnosis), fieldName, value, confidence (decimal 0-1), sourcePageNumber, sourceTextSnippet. Relationships: many-to-one with ClinicalDocument, Patient
- **DataConflict**: Represents conflicting data across documents for a patient. Attributes: id (UUID), patientId (FK), fieldName, value1, sourceDocumentId1 (FK), value2, sourceDocumentId2 (FK), resolutionStatus (Unresolved/Resolved/PendingReview), resolvedValue, resolvedBy (FK), resolvedAt. Relationships: many-to-one with Patient
- **MedicalCode**: Represents an AI-suggested or finalized medical code. Attributes: id (UUID), patientId (FK), codeType (ICD10/CPT), code, description, confidence (decimal 0-1), sourceDocumentId (FK), verificationStatus (Pending/Accepted/Rejected/Modified), verifiedBy (FK), verifiedAt. Relationships: many-to-one with Patient
- **AuditLog**: Immutable record of user actions. Attributes: id (UUID), userId (FK), action (Create/Read/Update/Delete), entityType, entityId, details (JSONB), ipAddress, timestamp. Relationships: many-to-one with User. Constraints: INSERT only, no UPDATE or DELETE permitted
- **Notification**: Represents a sent or pending notification. Attributes: id (UUID), patientId (FK), appointmentId (FK), channel (SMS/Email), templateType, status (Pending/Sent/Failed), sentAt, retryCount, lastRetryAt. Relationships: many-to-one with Patient, Appointment
- **InsuranceValidation**: Represents an insurance pre-check result. Attributes: id (UUID), patientId (FK), providerName, insuranceId, result (Matched/NotMatched), validatedAt. Relationships: many-to-one with Patient
- **QueueEntry**: Represents a position in the same-day queue. Attributes: id (UUID), patientId (FK), appointmentId (FK), position (integer), arrivalTime, status (Waiting/Called/Removed). Relationships: many-to-one with Patient; one-to-one with Appointment
- **CalendarSync**: Represents a synced calendar event. Attributes: id (UUID), patientId (FK), appointmentId (FK), provider (Google/Outlook), externalEventId, syncStatus (Synced/Failed/Revoked), syncedAt. Relationships: many-to-one with Patient, Appointment
- **NoShowRisk**: Represents a calculated risk score. Attributes: id (UUID), appointmentId (FK), score (decimal 0-1), factors (JSONB), calculatedAt. Relationships: one-to-one with Appointment
- **Specialty**: Represents a medical provider specialty. Attributes: id (UUID), name, description. Relationships: one-to-many with Appointment

## AI Requirements

### AI Functional Requirements

- AIR-001: System MUST extract structured patient data (vitals, medical history, medications, allergies, diagnoses) from uploaded PDF clinical documents using AI-powered document processing, producing JSON output with per-field confidence scores [traces to NFR-002, NFR-011]
- AIR-002: System MUST provide source citations (document name, page number, extracted text snippet) for every AI-extracted data point to support Trust-First verification [traces to NFR-011]
- AIR-003: System MUST fall back to manual staff review when AI extraction confidence falls below 80% for any individual data field [traces to NFR-011, NFR-018]
- AIR-004: System MUST conduct AI-powered conversational intake through multi-turn natural language interaction, extracting structured data (demographics, medical history, symptoms, medications) from patient responses [traces to NFR-001]
- AIR-005: System MUST suggest applicable ICD-10 diagnosis codes from aggregated patient data with confidence scores and source document references [traces to NFR-011]
- AIR-006: System MUST map and suggest applicable CPT procedure codes from clinical documentation with confidence scores and source document references [traces to NFR-011]
- AIR-007: System MUST calculate no-show risk scores using rule-based assessment factors (appointment history, lead time, reminder engagement) augmented by AI pattern recognition from historical appointment data [traces to NFR-011]
- AIR-008: System MUST process post-visit clinical notes to extract relevant patient data and update the consolidated patient record with source-linked extracted fields [traces to NFR-002]

### AI Quality Requirements

- AIR-Q01: System MUST maintain an AI-Human Agreement Rate exceeding 98% for suggested clinical data and medical codes, measured against staff-verified ground truth on evaluation datasets [traces to NFR-011]
- AIR-Q02: System MUST achieve p95 latency of 30 seconds or less for AI-powered document extraction and medical coding responses [traces to NFR-002]
- AIR-Q03: System MUST enforce output schema validity of 99% or higher for all AI-generated structured data (valid JSON, correct field types, required fields present) [traces to NFR-011]
- AIR-Q04: System MUST maintain a hallucination rate below 2% on clinical data extraction, measured by comparing AI outputs against staff-verified ground truth [traces to NFR-011]

### AI Safety Requirements

- AIR-S01: System MUST redact or de-identify PII from prompts before transmitting to external AI model providers that do not hold a HIPAA Business Associate Agreement [traces to NFR-013]
- AIR-S02: System MUST enforce document-level access control filtering in retrieval pipelines, ensuring AI responses only reference documents the requesting user is authorized to access [traces to NFR-006]
- AIR-S03: System MUST log all AI prompts and responses for audit purposes with a retention period matching HIPAA audit log requirements (minimum 7 years) [traces to NFR-009, DR-011]
- AIR-S04: System MUST implement content filtering to prevent AI-generated outputs from containing harmful, biased, or clinically inappropriate recommendations [traces to NFR-013]

### AI Operational Requirements

- AIR-O01: System MUST enforce a token budget of 8,000 tokens per individual AI request to control cost and latency [traces to NFR-002]
- AIR-O02: System MUST implement circuit breaker pattern for AI model provider failures, falling back to manual workflow after 3 consecutive failures within a 5-minute window [traces to NFR-018]
- AIR-O03: System MUST support AI model version configuration changes without application redeployment, effective within 5 minutes [traces to NFR-016]
- AIR-O04: System MUST track and report AI model usage metrics (token consumption, latency, error rates, confidence score distributions) for operational monitoring [traces to NFR-009]

### RAG Pipeline Requirements

- AIR-R01: System MUST chunk clinical documents into segments of 512 tokens with 10% overlap for embedding generation and vector storage [traces to DR-005, DR-006]
- AIR-R02: System MUST retrieve top-5 relevant document chunks with cosine similarity threshold of 0.7 or higher for patient context queries [traces to AIR-001, AIR-005]
- AIR-R03: System MUST re-rank retrieved chunks using semantic relevance scoring before constructing AI prompts to maximize extraction accuracy [traces to AIR-Q01]

### AI Architecture Pattern

**Selected Pattern:** Hybrid (RAG + Tool Calling + Conversational)

**Pattern Selection Rationale:**

| Pattern        | Applied To                                                                                                               | Rationale                                                                          |
| -------------- | ------------------------------------------------------------------------------------------------------------------------ | ---------------------------------------------------------------------------------- |
| RAG            | Clinical document extraction (AIR-001), 360° patient view aggregation (FR-046, FR-048), post-visit notes upload (FR-044) | Requires grounding in patient-specific uploaded documents with citation support    |
| Tool Calling   | ICD-10 code suggestion (AIR-005), CPT code mapping (AIR-006), insurance validation (FR-038)                              | Requires structured lookups against medical code databases and validation datasets |
| Conversational | AI intake (AIR-004)                                                                                                      | Multi-turn natural language dialogue with structured data extraction               |

**Trust-First Verification Flow:**

All AI-generated outputs follow a mandatory verification pipeline:

1. AI produces structured output with per-field confidence scores
2. Fields below 80% confidence are flagged for priority staff review
3. Staff receives AI output in a verification interface with source citations
4. Staff accepts, rejects, or modifies each AI-suggested data point
5. Only staff-verified data is committed to the patient's consolidated record
6. All verification actions are recorded in the immutable audit log

## Architecture and Design Decisions

- **AD-1: Modular Monolith over Distributed Microservices** — The platform is structured as a single deployable unit with clearly separated bounded modules (Auth, Patient, Appointment, Clinical, AI, Notification, Admin), enabling team autonomy and clear domain boundaries without the operational complexity of distributed services. This aligns with the free-tier hosting constraint and Phase 1 scope. Justified by NFR-003 (99.9% uptime without distributed tracing overhead), NFR-016 (internal module boundaries enable future extraction to microservices), and the free-tier infrastructure constraint (running 7+ independent services on free tiers is not viable)
- **AD-2: CQRS within Services** — Each service separates command (write) and query (read) paths using MediatR, enabling optimized read models for dashboards (staff scheduling, 360° patient view) without impacting write performance. Justified by NFR-001 (2-second API response) and NFR-010 (100 concurrent users)
- **AD-3: Event-Driven Async Processing** — Waitlist swap notifications, AI document processing, reminder scheduling, and calendar sync use asynchronous event-driven patterns to avoid blocking synchronous user flows. Justified by NFR-002 (AI latency tolerance) and NFR-018 (graceful degradation)
- **AD-4: API Gateway Pattern** — A single entry point handles cross-cutting concerns (authentication, rate limiting, request routing, correlation ID injection) before forwarding to downstream services. Justified by NFR-006 (RBAC), NFR-017 (rate limiting), and NFR-014 (input validation)
- **AD-5: Collocated Vector Store (pgvector)** — AI embeddings are stored in PostgreSQL via the pgvector extension rather than a separate vector database, eliminating additional infrastructure cost and operational complexity. Justified by the free-tier hosting constraint and DR-006 (extracted data storage)
- **AD-6: Semantic Kernel for AI Orchestration** — Microsoft Semantic Kernel provides a .NET-native abstraction layer for LLM interactions, enabling prompt management, model switching, and plugin-based tool calling without vendor lock-in. Justified by AIR-O02 (circuit breaker), AIR-O03 (model version management), and TR-002 (.NET ecosystem alignment)
- **AD-7: Immutable Append-Only Audit Log** — The audit log table permits INSERT operations only at the database level (no UPDATE or DELETE), with application-level enforcement via a write-only repository pattern. Justified by NFR-009, NFR-013 (HIPAA), and DR-011 (7-year retention)
- **AD-8: Redis-Backed Session and Cache Layer** — Upstash Redis handles session management (15-minute TTL), appointment slot availability caching, and real-time queue state. Justified by NFR-007 (session timeout), NFR-020 (slot staleness), and the serverless free-tier constraint
- **AD-9: JWT with Refresh Token Rotation** — Stateless JWT access tokens (short-lived, 15 minutes) paired with rotating refresh tokens provide secure authentication without server-side session state in the auth service. Redis tracks active sessions for forced logout and timeout enforcement. Justified by NFR-006, NFR-007, NFR-015

## Technical Constraints & Assumptions

### Constraints

- Phase 1 hosting is limited to free/open-source platforms exclusively; no AWS, Azure, or paid cloud services permitted
- Phase 1 excludes provider logins, payment gateway integration, family member profiles, patient self-check-in, direct EHR integration, and full claims submission
- Patient self-check-in is explicitly excluded; only Staff role can mark patients as "Arrived"
- Clinical document uploads are restricted to PDF format only in Phase 1
- Insurance pre-check validates against an internal dummy dataset, not real-time insurance provider APIs
- Free-tier hosting platforms impose resource limits (compute, storage, bandwidth) that may constrain concurrent user capacity beyond the 100-user target
- AI model provider API rate limits and token quotas on free/starter tiers may constrain throughput for document extraction and medical coding operations

### Assumptions

- OpenAI API provides sufficient free credits or a starter tier for Phase 1 development and demonstration; Azure OpenAI with HIPAA BAA will be adopted for production deployment
- Neon PostgreSQL free tier provides adequate storage and compute for Phase 1 data volumes (estimated under 10,000 patient records, under 50,000 appointments)
- SendGrid and Twilio free tiers provide sufficient monthly quotas for Phase 1 notification volumes
- PDF clinical documents uploaded by patients are machine-readable (text-based PDFs); scanned image PDFs with OCR requirements are handled on a best-effort basis with lower confidence scores
- The development team has proficiency in Angular 18, .net 10, and PostgreSQL; AI/ML integration via Semantic Kernel may require ramp-up time
- Google Calendar API and Microsoft Graph API free tiers will remain available and sufficient for Phase 1 calendar sync volumes

## Development Workflow

1. **Project Scaffolding** — Initialize Angular 18 workspace (standalone components, signals) and .net 10 solution with modular project structure (Auth, Patient, Appointment, Clinical, AI, Notification, Admin modules), Docker Compose for local orchestration, and PostgreSQL with pgvector extension
2. **Authentication and Authorization Service** — Implement JWT-based authentication with bcrypt/Argon2 password hashing, refresh token rotation, RBAC middleware, session management via Upstash Redis (15-minute TTL), and audit logging for auth events
3. **Patient Management Service** — Implement patient registration with email verification, profile CRUD, and role-based data access (patients access own records only, staff access authorized scope)
4. **Appointment Booking Service** — Implement slot management with real-time availability (Redis cache), booking CRUD, waitlist enrollment with FIFO swap logic, same-day queue management, appointment PDF generation (QuestPDF), and calendar sync integrations (Google, Outlook)
5. **Notification Service** — Implement configurable multi-channel reminder engine (SMS via Twilio, Email via SendGrid) with retry logic, delivery tracking, and graceful degradation on provider failure
6. **Clinical Intelligence Service** — Implement clinical document upload with PDF validation, AI-powered extraction pipeline (Semantic Kernel + OpenAI), RAG pipeline with pgvector, 360-degree patient view aggregation, data conflict detection, and staff verification workflow
7. **Medical Coding Service** — Implement ICD-10 and CPT code suggestion engine using AI extraction from aggregated patient data, staff review and finalization interface, and agreement rate tracking
8. **No-show Risk Assessment** — Implement rule-based risk scoring augmented by AI pattern analysis from historical appointment data, with scores displayed on staff scheduling dashboard
9. **Admin Panel** — Implement user management (create, update, deactivate across all roles), reminder configuration, and system-level settings
10. **Integration Testing and E2E Validation** — Execute end-to-end testing via Playwright covering all use cases (UC-001 through UC-018), validate AI agreement rate against test datasets, perform security testing (OWASP Top 10), and deploy to free-tier infrastructure
