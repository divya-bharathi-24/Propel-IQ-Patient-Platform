## 7. Content & Tone

### Voice & Tone

- **Overall Tone**: Calm, professional, trustworthy — healthcare-grade clarity without clinical coldness
- **Error Messages**: Specific, non-blaming, actionable (e.g., "We couldn't verify your insurance. You can still continue booking.")
- **Empty States**: Encouraging and action-oriented (e.g., "No appointments yet — book your first visit to get started")
- **Success Messages**: Brief and reassuring (e.g., "Appointment confirmed! Check your email for the PDF summary")
- **AI Suggestions**: Transparent with evidence cues (e.g., "Suggested based on: Visit Note — 2025-03-12")

### Content Guidelines

- **Headings**: Sentence case (e.g., "Book an appointment" not "Book An Appointment")
- **CTAs**: Action verbs — "Book appointment," "Upload document," "Verify profile," "Confirm codes"
- **Labels**: Concise, descriptive — no abbreviations (e.g., "Date of birth" not "DOB")
- **Placeholder Text**: Illustrative examples — "e.g., Blue Shield of California" not "Enter insurance name"
- **HIPAA Fields**: Masked by default — user must explicitly click to reveal (e.g., insurance ID, DOB)

## 8. Data & Edge Cases

### Data Scenarios

| Scenario | Description | Handling |
|----------|-------------|----------|
| No appointments | New patient with zero bookings | Empty state with "Book your first appointment" CTA |
| No documents uploaded | Patient has not uploaded clinical documents | Empty state explaining the benefit + "Upload documents" CTA |
| AI extraction in progress | Documents uploaded but extraction not complete | "Processing" badge on dashboard; skeleton in 360-degree view |
| AI extraction failed | Document could not be parsed | Per-document error badge; staff notified; exclude from aggregation |
| No conflicts detected | All documents consistent | 360-degree view renders normally without conflict section |
| Conflicting medications | Two documents with different dosages | Conflict section visible; resolution required before verification |
| Queue empty | No same-day appointments | Empty state with "No patients in queue today" and walk-in button |
| Large document set | Patient has 20 uploaded documents | Paginated document list; 360-degree view grouped by clinical category |
| Insurance not recognized | Insurance name/ID not in dummy record set | "Not Recognized" badge; non-blocking — booking proceeds |
| Slot swap occurred | Patient's preferred slot opened | In-app toast; email/SMS sent; dashboard appointment updated |
| Session timeout | 15 minutes of inactivity | Warning modal at 13 minutes; auto-logout at 15 minutes |

### Edge Cases

| Case | Screen(s) Affected | Solution |
|------|--------------------|----------|
| Long patient name (>40 chars) | SCR-014, SCR-016, SCR-003 | Truncate to 40 chars with full name tooltip on hover |
| Long medication name | SCR-016, SCR-017 | Truncate with expand option |
| Multiple slot swap waitlists | SCR-003, SCR-005b | Each preferred slot shown as a separate waitlist card |
| Missing profile photo | SCR-010, SCR-014 | Avatar with initials fallback (first + last initial) |
| Document filename > 50 chars | SCR-011 | Truncate filename in list; full name in tooltip |
| Admin deactivates own account | SCR-020 | Block action with error: "You cannot deactivate your own account" |
| Network error during upload | SCR-011 | Per-file retry option; successfully uploaded files not re-uploaded |
| PDF confirmation email fails | SCR-007 | Toast warning; "Resend confirmation email" button on appointment detail |

## 11. Prototype Flows

### Flow: FL-001 — Patient Registration & Appointment Booking

**Flow ID**: FL-001
**Derived From**: UC-001
**Personas Covered**: Patient (Alex)
**Description**: New patient self-registers, verifies email, books an appointment with optional preferred slot, and receives confirmation.

#### Flow Sequence

```text
1. Entry: SCR-001 / Default
   - Trigger: User clicks "Create account"
   |
   v
2. SCR-002 / Default
   - Action: User fills registration form (name, email, password, DOB)
   |
   v
3. SCR-002a / Default
   - Action: System sends verification email; user verifies
   |
   v
4. SCR-003 / Empty (no appointments yet)
   - Action: User clicks "Book appointment"
   |
   v
5. SCR-005 / Default (slot selection)
   - Action: User selects date and available slot
   |
   v
6. SCR-005b / Default (preferred slot)  [OPTIONAL BRANCH]
   - Action: User designates a preferred unavailable slot
   |
   v
7. SCR-006 / Default (insurance pre-check)
   - Action: User enters insurance name and member ID
   |
   +-- Verified -> SCR-006 / Verified state -> continue
   +-- Not Recognized -> SCR-006 / Not Recognized state -> continue (non-blocking)
   |
   v
8. SCR-008 or SCR-009 (intake mode selection)
   - Action: User selects AI or manual intake; begins intake
   |
   v
9. SCR-007 / Default (booking confirmation)
   - Action: System generates PDF and sends email; user offered calendar sync
   |
   v
10. SCR-012 / Default (calendar sync) [OPTIONAL]
    - Exit: Patient added to SCR-003 dashboard with appointment card
```

#### Required Interactions

- Slot selection grid: hover highlights; selected slot shows confirmation chip
- Intake mode toggle: switches between SCR-008 and SCR-009 preserving entered data
- Booking confirmation: "Add to Calendar" button triggers SCR-012 overlay
