### Story Format
> **As a** [role], **I want to** [capability], **so that** [benefit].

### Acceptance Criteria Format
> **Given** [precondition], **When** [action], **Then** [outcome].

Each story is validated against: **I**ndependent · **N**egotiable · **V**aluable · **E**stimable · **S**mall · **T**estable

### US-TECH-001 · Full-Stack Project Scaffolding

**Story**
> As a **DevOps engineer**, I want the Angular 18 SPA and .NET 9 modular monolith to be fully scaffolded with all seven domain modules, so that every feature team can begin development immediately on a consistent, convention-aligned codebase.

**Business Value**
Without scaffolding, every developer independently configures the project, creating divergence. This story eliminates that risk and enables all feature epics to start in parallel.

**Acceptance Criteria**

| # | Given | When | Then |
|---|---|---|---|
| AC1 | An empty repository | A developer runs `ng new` and the backend `dotnet new` scaffold commands | Angular 18 workspace with standalone components and NgRx Signals, and a .NET 9 solution with 7 module projects (Auth, Patient, Appointment, Clinical, AI, Notification, Admin) are created |
| AC2 | The scaffold is applied | A developer runs `dotnet build` | Build succeeds with zero errors and zero warnings |
| AC3 | The scaffold is applied | A developer runs `ng build` | Angular build succeeds and outputs a production bundle |
| AC4 | The scaffold is applied | Swagger UI is opened at `/swagger` | All module API endpoint groups are listed with OpenAPI 3.0 documentation |
| AC5 | The scaffold is applied | Serilog is configured | Every HTTP request emits a structured log entry with a correlation ID |

**Story Points**: 8 · **Priority**: P1 · **Req Traceability**: TR-001, TR-002, TR-006, TR-018
**Dependencies**: None
**INVEST Notes**: Foundational enabler story; not independently deliverable to end users but enables all feature stories.

### US-DATA-001 · Domain Entity Data Model

**Story**
> As a **DevOps engineer**, I want all 17 domain entities (Patient, User, Appointment, WaitlistEntry, IntakeRecord, ClinicalDocument, ExtractedData, DataConflict, MedicalCode, AuditLog, Notification, InsuranceValidation, QueueEntry, CalendarSync, NoShowRisk, Specialty, and a pgvector embedding column on ExtractedData) to be implemented with EF Core fluent configuration, so that every feature epic has a validated, constraint-enforced data layer to build against.

**Business Value**
A correct data model is the foundation for all HIPAA-compliant data handling. Incorrect schemas discovered late cause costly rework and compliance risk.

**Acceptance Criteria**

| # | Given | When | Then |
|---|---|---|---|
| AC1 | A fresh PostgreSQL instance | `dotnet ef database update` runs | All 17 entity tables are created with correct column types and constraints |
| AC2 | Patient and Appointment entities exist | A Patient record is soft-deleted | The record status is set to Deactivated; the record remains queryable |
| AC3 | ExtractedData entity exists | An embedding is inserted via pgvector | The embedding is stored and a cosine similarity query returns results |
| AC4 | IntakeRecord entity exists | A record is created with JSON demographics | The JSONB column stores and retrieves the structured data without data loss |
| AC5 | All FK relationships are configured | A Patient referenced by an Appointment is queried | EF Core returns the related Appointment via navigation property |
| AC6 | Specialty seed script runs | Seed data is applied | At least 5 specialty records and 10 dummy insurer records exist |

**Story Points**: 13 · **Priority**: P1 · **Req Traceability**: DR-001–DR-018, TR-003, TR-008
**Dependencies**: US-TECH-001, US-TECH-003
**INVEST Notes**: Large story; complex data model requires thorough testing. Cannot be meaningfully split without losing entity relationship context.

### US-DATA-002 · Immutable Audit Log Enforcement

**Story**
> As a **DevOps engineer**, I want the AuditLog table to be INSERT-only at the database level — with UPDATE and DELETE operations blocked by a PostgreSQL policy or trigger — so that the platform satisfies HIPAA's requirement for tamper-proof audit records.

**Business Value**
HIPAA requires immutable access logs. Any loophole allowing modification or deletion creates regulatory liability and undermines forensic integrity.

**Acceptance Criteria**

| # | Given | When | Then |
|---|---|---|---|
| AC1 | The AuditLog table exists | An application attempts an `UPDATE` on any audit log row via SQL | The database returns an error and the update is rejected |
| AC2 | The AuditLog table exists | An application attempts a `DELETE` on any audit log row via SQL | The database returns an error and the delete is rejected |
| AC3 | The AuditLog table exists | A new audit event is written via the application's write-only repository | The INSERT succeeds and the record is queryable |
| AC4 | Audit records exist | The records are 7 years old (simulated via timestamp) | The records are still present and queryable |

**Story Points**: 5 · **Priority**: P1 · **Req Traceability**: DR-011, NFR-009, NFR-013
**Dependencies**: US-DATA-001
**INVEST Notes**: Independently testable via direct SQL; HIPAA-critical.

### US-001-002 · Role-Based Access Control

**Story**
> As a **platform administrator**, I want every API endpoint to enforce role-based access control at the gateway level, so that patients, staff, and admins can only access the resources and actions permitted for their role — preventing unauthorized data access and privilege escalation.

**Business Value**
RBAC is the primary HIPAA access control mechanism. Without it, a patient could access clinical records or an adversary could escalate privileges.

**Acceptance Criteria**

| # | Given | When | Then |
|---|---|---|---|
| AC1 | A Patient JWT is valid | The patient calls a Staff-only endpoint (e.g., `POST /walk-in-bookings`) | HTTP 403 is returned; no data is exposed |
| AC2 | A Staff JWT is valid | The staff member calls an Admin-only endpoint (e.g., `GET /audit-logs`) | HTTP 403 is returned |
| AC3 | No JWT is present | An unauthenticated request calls any authenticated endpoint | HTTP 401 is returned |
| AC4 | An expired JWT is presented | Any authenticated request is made | HTTP 401 is returned; a new JWT must be obtained via refresh |
| AC5 | A Patient JWT is valid | The patient calls their own profile endpoint (`GET /patients/{own-id}`) | HTTP 200 is returned with their own data |
| AC6 | A Patient JWT is valid | The patient calls another patient's profile endpoint | HTTP 403 is returned |

**Story Points**: 8 · **Priority**: P1 · **Req Traceability**: FR-001, NFR-006, NFR-013, TR-007

### US-001-003 · Session Inactivity Timeout

**Story**
> As a **platform user** (Patient, Staff, or Admin), I want my session to automatically end after exactly 15 minutes of inactivity and redirect me to the login page, so that unattended sessions do not expose PHI to unauthorized individuals.

**Business Value**
Automatic session termination is a HIPAA Security Rule control (§ 164.312(a)(2)(iii)). Unattended workstations in clinical settings represent a critical access control risk.

**Acceptance Criteria**

| # | Given | When | Then |
|---|---|---|---|
| AC1 | A Patient is authenticated | No API activity occurs for 15 minutes | The next API request returns HTTP 401 and the frontend redirects to the login page |
| AC2 | A Staff member is authenticated | The Staff member makes an API call 14 minutes after their last activity | The request succeeds and the inactivity timer resets |
| AC3 | A session timeout occurs | The patient logs in again | A new session is created; the previous session token is invalid |
| AC4 | Redis is unavailable | A session timeout check is performed | The system defaults to enforcing timeout (fail-secure); re-authentication is required |

**Story Points**: 3 · **Priority**: P1 · **Req Traceability**: FR-004, NFR-007, TR-005
**Dependencies**: US-001-002, US-TECH-001
**INVEST Notes**: Small, precisely specifiable, independently testable via timer simulation.

### US-001-004 · Authentication Event Audit Logging

**Story**
> As a **compliance officer**, I want every authentication event (login, logout, failed attempt, session timeout) to be written to the immutable audit log with a UTC timestamp, user ID, role, and originating IP address, so that a complete access history is available for HIPAA compliance audits.

**Business Value**
HIPAA requires organizations to maintain records of who accessed PHI systems and when. Authentication events are the first layer of this requirement.

**Acceptance Criteria**

| # | Given | When | Then |
|---|---|---|---|
| AC1 | A patient logs in successfully | The login request completes | An AuditLog entry is created: event=Login, userId, role=Patient, IP, UTC timestamp |
| AC2 | A user submits an incorrect password | The login request completes | An AuditLog entry is created: event=LoginFailed, IP, UTC timestamp (userId is null if account not found) |
| AC3 | An authenticated user logs out | The logout request completes | An AuditLog entry is created: event=Logout, userId, role, IP, UTC timestamp |
| AC4 | A session times out | The timeout is enforced | An AuditLog entry is created: event=SessionTimeout, userId, role, UTC timestamp |
| AC5 | Audit entries are created | An attempt is made to DELETE an audit log entry directly via SQL | The operation is rejected by the database policy |

**Story Points**: 3 · **Priority**: P1 · **Req Traceability**: FR-006, NFR-009, NFR-015
**Dependencies**: US-DATA-002, US-001-002
**INVEST Notes**: Self-contained log writer; testable by querying audit table after each auth event.

### US-001-005 · Secure Password Storage & Complexity

**Story**
> As a **patient**, I want my password to be stored using bcrypt or Argon2 hashing and rejected at registration if it does not meet complexity requirements, so that my account is protected against credential theft even in the event of a database breach.

**Business Value**
Plaintext or weakly hashed passwords are the most exploited credential vulnerability. bcrypt/Argon2 hashing ensures passwords are computationally infeasible to reverse.

**Acceptance Criteria**

| # | Given | When | Then |
|---|---|---|---|
| AC1 | A patient registers with a valid password | The account is created | The `passwordHash` column in the User table contains a bcrypt or Argon2 hash — never plaintext |
| AC2 | A database administrator queries the User table directly | They read the `passwordHash` column | The value is a hash string; the original password cannot be recovered |
| AC3 | A patient registers | Their password is "password" (no complexity) | HTTP 422 is returned with a detailed validation error |
| AC4 | A patient registers | Their password is "P@ssw0rd!" (compliant) | Registration proceeds |

**Story Points**: 2 · **Priority**: P1 · **Req Traceability**: NFR-008
**Dependencies**: US-001-001
**INVEST Notes**: Simple, single-concern story. Verified by DB inspection and registration API tests.

### US-001-006 · Rate Limiting on Auth Endpoints

**Story**
> As a **security engineer**, I want the `/auth/*` endpoints to enforce rate limiting, so that brute-force credential attacks are blocked before they can enumerate valid accounts or exhaust system resources.

**Business Value**
Unprotected auth endpoints are the primary attack surface for credential stuffing. Rate limiting provides a non-negotiable baseline defence per OWASP A07.

**Acceptance Criteria**

| # | Given | When | Then |
|---|---|---|---|
| AC1 | A client IP sends login requests | More than 10 failed login attempts occur within 60 seconds from the same IP | Subsequent requests return HTTP 429 with a `Retry-After` header |
| AC2 | A rate limit is applied | The configured window expires | New requests from the same IP are accepted again |
| AC3 | A legitimate user is rate limited | They wait for the retry window | They can log in successfully after the window expires |
| AC4 | Rate limiting is active | A successful login occurs within the window | The success does not reset the failed-attempt counter for abuse detection |

**Story Points**: 3 · **Priority**: P1 · **Req Traceability**: NFR-017
**Dependencies**: US-TECH-001
**INVEST Notes**: Independently implementable as middleware; testable with repeated HTTP calls.

| AC4 | The Redis cache is cold | The first request is made | The API fetches from the database and populates the cache; response completes within 2 seconds |

**Story Points**: 5 · **Priority**: P1 · **Req Traceability**: FR-011, NFR-020, NFR-001
**Dependencies**: US-001-002, US-DATA-001

## Appendix B — INVEST Validation Summary

| Principle | How Enforced |
|---|---|
| **Independent** | Stories within an epic avoid hard cross-epic dependencies where possible; only foundational stories (TECH, DATA) are pre-conditions |
| **Negotiable** | Acceptance criteria describe *what* not *how*; implementation approach remains open |
| **Valuable** | Each story includes a Business Value section tied to a measurable platform outcome (no-show rate, prep time, HIPAA compliance) |
| **Estimable** | Story points use Fibonacci scale; stories > 13 points are flagged for decomposition |
| **Small** | Stories are scoped to complete in a single sprint; US-DATA-001 and US-008I-003 are the largest (13 pts) and represent irreducible pipeline units |
| **Testable** | Every story has Given/When/Then acceptance criteria; all ACs map to test cases in `test-plan.md` |
