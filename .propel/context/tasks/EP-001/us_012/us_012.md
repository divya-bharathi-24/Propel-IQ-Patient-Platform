# User Story - US_012

## Story ID

- ID: US_012

## Story Title

- JWT Authentication with Role-Based Access

## Description

- As a registered user (Patient, Staff, or Admin), I want to log in with my email and password and receive role-appropriate access to the platform via JWT authentication, so that I can securely access only the features authorized for my role.

## Acceptance Criteria

1. **Given** I am on the login page, **When** I enter valid credentials, **Then** the system authenticates me, issues a JWT access token (15-minute expiry) and a rotating refresh token, and redirects me to my role-specific dashboard.

2. **Given** I am authenticated, **When** I make an API request, **Then** the system validates my JWT token and enforces role-based access control, restricting endpoints based on my role (Patient, Staff, Admin).

3. **Given** I have a valid refresh token, **When** my access token expires, **Then** the system issues a new access token via the refresh token endpoint without requiring re-login.

4. **Given** I am on the login page, **When** I enter invalid credentials, **Then** the system displays a generic error message "Invalid email or password" without revealing whether the email exists, and increments the failed login counter.

5. **Given** my account has been deactivated, **When** I attempt to log in, **Then** the system displays "Contact administrator" and denies access.

6. **Given** I am authenticated, **When** I access an endpoint restricted to a different role, **Then** the system returns a 403 Forbidden response.

## Edge Cases

- What happens when the refresh token is reused after rotation? (Detect token reuse as potential compromise; invalidate all tokens for the user session and force re-login)
- How does the system handle concurrent logins from multiple devices? (Each device receives independent session; all sessions are tracked in Redis for forced logout capability)
