# User Story - US_013

## Story ID

- ID Format: US_013

## Story Title

- Authentication Event Audit Logging

## Description

- As a compliance officer, I want every authentication event (login, logout, failed attempt, session timeout) to be recorded in the immutable audit log with a UTC timestamp, user ID, role, and originating IP address, so that all access events are traceable for HIPAA audit requirements.

## Acceptance Criteria

1. **Given** a user logs in successfully, **When** the authentication event fires, **Then** an AuditLog record is inserted with `action = Login`, `userId`, `role`, `ipAddress`, and a UTC `timestamp` — and no UPDATE or DELETE is possible on this record.

2. **Given** a login attempt fails (wrong password), **When** the error is returned, **Then** an AuditLog record is inserted with `action = FailedLogin`, the attempted email (not password), `ipAddress`, and UTC timestamp.

3. **Given** a session auto-expires after 15 minutes of inactivity, **When** the TTL fires in Redis, **Then** an AuditLog record is inserted with `action = SessionTimeout`, `userId`, and UTC timestamp within 5 seconds of the timeout event.

4. **Given** a user explicitly logs out, **When** the logout endpoint is called, **Then** an AuditLog record is inserted with `action = Logout` before the session is invalidated.

## Edge Cases

- What happens if the audit log write fails (DB unavailable)? (Audit log write failures are retried once; if still failing, the authentication action succeeds but a critical alert is raised — security over audit atomicity)
- What happens when the same IP triggers more than 10 failed logins in 5 minutes? (Rate limiter blocks further attempts per NFR-017; the block event itself is logged as `RateLimitBlock`)
