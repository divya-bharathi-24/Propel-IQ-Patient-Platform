# User Story - US_014

## Story ID

- ID Format: US_014

## Story Title

- Rate Limiting, Input Validation & Encryption Controls

## Description

- As a security engineer, I want rate limiting on all public auth endpoints, FluentValidation on all inputs, Argon2 password hashing, TLS 1.2+ in transit, and AES-256 at rest enforced at the infrastructure level, so that the platform is hardened against injection, brute-force, and data-at-rest attacks from day one.

## Acceptance Criteria

1. **Given** an unauthenticated caller sends more than 10 requests to `/api/auth/login` within 60 seconds, **When** the rate limiter triggers, **Then** subsequent requests receive HTTP 429 Too Many Requests with a `Retry-After` header and the event is logged.

2. **Given** an API endpoint receives a request with an invalid or missing required field, **When** FluentValidation runs, **Then** the request is rejected with HTTP 400 and a structured error response listing each violated field — no internal stack trace is exposed.

3. **Given** a patient password is stored, **When** I inspect the database, **Then** the stored value is an Argon2 hash (never plaintext), and the hash includes a unique salt.

4. **Given** all data is in transit between client and server, **When** I inspect the TLS handshake, **Then** only TLS 1.2 or higher is negotiated; TLS 1.0/1.1 connections are rejected.

## Edge Cases

- What happens if an attacker submits a payload with excessively long strings? (FluentValidation enforces `MaxLength` on all string fields; oversized input is rejected before processing)
- What happens if the rate limiter configuration is misconfigured and blocks all traffic? (Integration test validates the rate limiter allows legitimate traffic; alert fires if >50% of requests are rejected within 1 minute)
