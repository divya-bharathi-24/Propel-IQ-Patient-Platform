# User Story - US_015

## Story ID

- ID Format: US_015

## Story Title

- Patient Profile View & Structured Demographic Edit

## Description

- As an authenticated Patient, I want to view my full profile and edit non-locked fields (contact info, emergency contact, communication preferences) without staff assistance, so that my personal information stays accurate and up to date.

## Acceptance Criteria

1. **Given** I am authenticated as a Patient, **When** I navigate to my profile page, **Then** I see my legal name, date of birth, biological sex, email, phone number, address, insurance details (insurer name, member ID, group number), and emergency contact displayed.

2. **Given** I edit a non-locked field (e.g., phone number or communication preference), **When** I save the changes, **Then** the updated value is persisted, a success confirmation is shown, and a patient data modification event is logged in the audit trail.

3. **Given** I attempt to edit a locked field (e.g., legal name or date of birth), **When** I view the form, **Then** those fields are read-only with a note indicating staff assistance is required for changes.

4. **Given** I submit a profile edit with an invalid phone number format, **When** the form is validated, **Then** an inline error message specifies the expected format without blocking edits to other fields.

## Edge Cases

- What happens if my session expires while I am editing my profile? (Unsaved changes are preserved in sessionStorage; after re-authentication, I am returned to the edit form with data pre-populated)
- What happens if two browser tabs save conflicting profile edits simultaneously? (Last-write-wins with optimistic concurrency; a stale data warning is shown if conflict is detected)
