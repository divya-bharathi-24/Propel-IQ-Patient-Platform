# User Story - US_015

## Story ID

- ID: US_015

## Story Title

- Appointment Booking with Slot Display

## Description

- As an authenticated patient, I want to book an appointment by selecting a provider specialty and preferred date with real-time available slot display, so that I can efficiently schedule healthcare visits with up-to-date availability.

## Acceptance Criteria

1. **Given** I am authenticated as a Patient, **When** I select a provider specialty and date range, **Then** the system displays available time slots filtered by my criteria within 2 seconds (p95 latency).

2. **Given** available slots are displayed, **When** I select a time slot and confirm booking, **Then** the system reserves the slot, creates an appointment record with status "Booked", and displays a booking confirmation with appointment summary.

3. **Given** available slots are displayed, **When** another patient books the same slot before I confirm, **Then** the system displays a conflict message and automatically refreshes the available slots.

4. **Given** the slot availability is cached, **When** a slot is booked or released, **Then** the Redis cache is updated within 5 seconds to reflect current availability (NFR-020).

5. **Given** I select a specialty and date range, **When** no available slots exist, **Then** the system suggests alternative dates or offers waitlist enrollment for preferred unavailable slots.

## Edge Cases

- What happens when I attempt to book during a network interruption? (Client displays offline indicator; booking is retried upon reconnection with slot re-validation)
- How does the system handle simultaneous booking attempts for the last available slot? (Optimistic locking ensures only one booking succeeds; the other receives a 409 Conflict)
