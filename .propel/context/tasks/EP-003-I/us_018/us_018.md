# User Story - US_018

## Story ID

- ID Format: US_018

## Story Title

- Real-Time Slot Availability with Redis Cache

## Description

- As an authenticated Patient, I want to see currently available appointment slots in real time with no more than 5-second staleness, so that I can select a slot with confidence it is still available when I book.

## Acceptance Criteria

1. **Given** I navigate to the slot selection step, **When** the available slots are fetched, **Then** the list reflects confirmed and pending bookings with a cache staleness of ≤5 seconds from the last slot state change.

2. **Given** the slot list is loaded from Redis cache, **When** a booking or cancellation occurs, **Then** the Redis cache is invalidated and the next request returns the updated slot list within 5 seconds.

3. **Given** the Redis cache is unavailable, **When** I request slots, **Then** the system falls back to a direct PostgreSQL query, returns the slot list successfully, and logs a cache miss warning.

4. **Given** all slots for a given day are booked, **When** I view the slot picker, **Then** fully booked days are greyed out with a "No slots available" label and a prompt to join the waitlist.

## Edge Cases

- What happens if two patients both see a slot as available and both attempt to book simultaneously? (Optimistic locking on the Appointment record per FR-013; the second attempt receives a "Slot no longer available" error and is prompted to choose another)
- What happens if the slot cache returns stale data showing an already-booked slot as available? (Optimistic locking catches the conflict at commit time; user is shown the updated availability)
