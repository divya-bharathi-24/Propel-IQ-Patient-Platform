# User Story - US_024

## Story ID

- ID Format: US_024

## Story Title

- Automatic Slot Swap Transaction Execution

## Description

- As a Patient on the waitlist for a preferred slot, I want the system to automatically swap me to my preferred slot within 60 seconds of it becoming available, so that I get my desired appointment without any manual intervention.

## Acceptance Criteria

1. **Given** a cancellation event occurs for a slot that is someone's preferred slot, **When** the event-driven monitor fires, **Then** the system queries WaitlistEntry for all matching `preferredSlot` waitlist entries ordered by `enrolledAt` ASC and selects the first eligible patient.

2. **Given** an eligible waitlisted patient is identified, **When** the atomic swap transaction executes, **Then** the current appointment is cancelled, the patient is assigned to the preferred slot as a new Appointment, the WaitlistEntry is marked `Swapped`, and the original slot is released — all within 60 seconds and inside a single database transaction.

3. **Given** the swap transaction is committed, **When** the Redis cache is invalidated, **Then** the slot availability cache reflects the new state within 5 seconds.

4. **Given** the preferred slot is claimed by another patient (race condition) before the swap executes, **When** the transaction fails, **Then** the WaitlistEntry remains `Active`, the next eligible patient in the FIFO queue is evaluated, and the original booking is unchanged.

## Edge Cases

- What happens if the database transaction deadlocks during the swap? (Deadlock retry with exponential backoff up to 3 attempts; on persistent failure, the WaitlistEntry remains Active and an alert is raised)
- What happens if the waitlisted patient's current appointment was already cancelled before the swap? (System verifies the current appointment status before the swap; if already cancelled, the WaitlistEntry is expired and the next queue entry is processed)
