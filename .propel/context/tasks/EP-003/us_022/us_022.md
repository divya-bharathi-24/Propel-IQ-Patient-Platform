# User Story - US_022

## Story ID

- ID: US_022

## Story Title

- Insurance Pre-Check Validation

## Description

- As a patient or staff member, I want to validate insurance provider name and insurance ID against an internal dummy dataset during booking or intake, so that insurance coverage can be soft-verified without blocking the workflow.

## Acceptance Criteria

1. **Given** I am in the booking or intake workflow, **When** I enter my insurance provider name and insurance ID, **Then** the system searches the internal predefined dummy insurance dataset for a match.

2. **Given** a matching insurance record is found, **When** the validation completes, **Then** the system displays "Insurance Verified" status to me and stores the validation result as Matched.

3. **Given** no matching record is found, **When** the validation completes, **Then** the system displays "Insurance Not Verified" status but does NOT block the booking or intake workflow from continuing.

4. **Given** I am in the booking workflow, **When** I choose to skip the insurance entry step, **Then** the system allows me to proceed without insurance data and the booking continues normally.

5. **Given** the insurance validation is performed, **When** the result is stored, **Then** an InsuranceValidation record is created with provider name, insurance ID, result (Matched/NotMatched), and timestamp.

## Edge Cases

- What happens when the insurance provider name has a slight misspelling? (Exact match only in Phase 1; system displays "Not Verified" with guidance to check spelling)
- How does the system handle duplicate insurance validation requests for the same appointment? (Latest validation result overwrites previous; history preserved via separate records)
