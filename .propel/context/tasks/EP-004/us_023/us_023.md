# User Story - US_023

## Story ID

- ID: US_023

## Story Title

- Staff Walk-in Booking Creation

## Description

- As a Staff member, I want to create walk-in bookings by searching for existing patients or optionally creating new patient accounts, so that walk-in patients can be efficiently booked into available slots or added to the same-day queue.

## Acceptance Criteria

1. **Given** I am authenticated as Staff, **When** I select the walk-in booking option, **Then** the system displays a patient search interface allowing search by name, email, or phone number.

2. **Given** I search for a patient, **When** a matching patient record is found, **Then** the system displays the patient details and allows me to proceed with booking.

3. **Given** I search for a patient, **When** no matching record is found, **Then** the system offers to create a new patient account with basic information (name, phone, email).

4. **Given** I have identified or created the patient, **When** I select a provider specialty and assign an available slot, **Then** the system creates the appointment record with createdBy set to my staff user ID.

5. **Given** the walk-in booking is created and the patient has an email address, **When** the booking is confirmed, **Then** the system sends an appointment confirmation email to the patient.

6. **Given** no available slots exist, **When** I attempt to create a walk-in booking, **Then** the system suggests adding the patient to the same-day queue instead.

## Edge Cases

- What happens when a patient insists they are not a staff member but tries to create a walk-in? (API enforces Staff-only access; Patient role receives 403 Forbidden)
- How does the system handle a walk-in for a patient with a deactivated account? (Staff can reactivate via admin workflow or create booking linked to deactivated profile with note)
