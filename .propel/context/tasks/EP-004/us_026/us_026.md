# User Story - US_026

## Story ID

- ID Format: US_026

## Story Title

- Staff Walk-In Booking with Optional Patient Account Creation

## Description

- As an authenticated Staff member, I want to create a walk-in appointment for a patient (with an optional quick patient account creation flow) so that unscheduled visits are properly registered and anonymous visits can be tracked via a temporary visit ID.

## Acceptance Criteria

1. **Given** I am authenticated as Staff and navigate to the walk-in booking interface, **When** I search for an existing patient, **Then** I see matching results by name or date of birth and can link the walk-in to an existing Patient record.

2. **Given** no matching patient is found, **When** I choose to create a basic profile, **Then** I enter name, contact number, and email — a Patient record is created and linked to the new walk-in Appointment.

3. **Given** no matching patient is found and I skip account creation, **When** the walk-in is confirmed, **Then** an Appointment record is created with a temporary anonymous visit ID and no `patientId` FK, and the visit appears in the same-day queue.

4. **Given** a Patient-role user attempts to access the walk-in booking endpoint, **When** the request is evaluated, **Then** HTTP 403 Forbidden is returned and no appointment is created.

## Edge Cases

- What happens if I create a Patient account with a duplicate email during walk-in? (System detects the duplicate and prompts me to link to the existing patient record instead)
- What happens if I try to book a walk-in for a time slot that is fully booked? (Walk-in is added to the same-day queue without a specific time slot; a QueueEntry with `position = next in queue` is created)
