# User Story - US_030

## Story ID

- ID: US_030

## Story Title

- Outlook Calendar Sync

## Description

- As a patient, I want to sync my booked appointments to my Outlook Calendar via Microsoft Graph API and OAuth 2.0, so that my healthcare appointments integrate with my Microsoft calendar and I receive native Outlook reminders.

## Acceptance Criteria

1. **Given** I have a booked appointment, **When** I select "Sync to Outlook Calendar", **Then** the system initiates the Microsoft OAuth 2.0 authorization flow if I have not previously authorized.

2. **Given** I authorize the application, **When** the OAuth flow completes, **Then** the system securely stores the access/refresh tokens and creates a calendar event via Microsoft Graph API v1.0 with appointment date, time, provider specialty, and location.

3. **Given** I have previously authorized Outlook Calendar access, **When** I sync a new appointment, **Then** the system skips the authorization step and directly creates the calendar event.

4. **Given** the calendar event is created, **When** the sync completes, **Then** the system creates a CalendarSync record with provider "Outlook", external event ID, syncStatus "Synced", and displays confirmation to me.

5. **Given** the Microsoft Graph API rate limit is exceeded, **When** the sync fails, **Then** the system queues the request and retries with exponential backoff, logging the delay.

6. **Given** my authorization is revoked or expired, **When** I attempt to sync, **Then** the system prompts me to re-authorize calendar access.

## Edge Cases

- What happens when both Google and Outlook sync are attempted for the same appointment? (Both syncs are independent; separate CalendarSync records created for each provider)
- How does the system handle Microsoft tenant restrictions? (OAuth flow respects tenant configuration; error message guides patient if their organization blocks third-party apps)
