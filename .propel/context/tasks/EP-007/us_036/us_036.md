# User Story - US_036

## Story ID

- ID Format: US_036

## Story Title

- Microsoft Outlook Calendar OAuth 2.0 Integration

## Description

- As a Patient, I want to sync my confirmed appointment to Microsoft Outlook Calendar via Microsoft Graph API OAuth 2.0, so that the appointment appears in my Outlook calendar automatically.

## Acceptance Criteria

1. **Given** I click "Add to Outlook Calendar" on my booking confirmation or dashboard, **When** I complete the Microsoft OAuth 2.0 flow, **Then** a calendar event is created via Microsoft Graph API (`POST /me/events`) with appointment date, start/end time, provider specialty, clinic name, appointment type, and booking reference number.

2. **Given** the Outlook event is successfully created, **When** the API response confirms creation, **Then** a CalendarSync record is stored with `provider = Outlook`, `externalEventId`, `syncStatus = Synced`, and an Outlook web event link is shown.

3. **Given** I choose "Download ICS" instead of connecting a calendar service, **When** the download is initiated, **Then** a valid `.ics` file is generated with all required appointment fields and downloaded to my device.

4. **Given** the Microsoft Graph API is unavailable, **When** my sync attempt fails, **Then** `syncStatus = Failed` is stored, a retry is scheduled, and the ICS download option is prominently shown as a fallback.

## Edge Cases

- What happens when a user has multiple Outlook calendars (personal, work)? (Event is added to the default/primary calendar; a follow-up UI option to choose a specific calendar is not in scope for Phase 1)
- What happens if the Microsoft OAuth consent is revoked after initial sync? (Next sync event returns 401; CalendarSync record is updated to `syncStatus = Revoked`; user is prompted to re-authorize)
