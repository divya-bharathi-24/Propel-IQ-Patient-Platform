# User Story - US_038

## Story ID

- ID Format: US_038

## Story Title

- Patient Clinical Document Upload with Encrypted Storage

## Description

- As an authenticated Patient, I want to upload my historical clinical PDF documents (up to 20 files, 25 MB each) through my dashboard, with all files encrypted at rest (AES-256) and in transit (TLS 1.2+), so that my sensitive medical records are stored securely.

## Acceptance Criteria

1. **Given** I navigate to the document upload interface from my dashboard, **When** I select up to 20 PDF files (each ≤25 MB), **Then** all files are validated client-side for type (PDF only) and size before upload begins.

2. **Given** the files pass client-side validation, **When** they are uploaded to the server, **Then** each file is validated server-side, encrypted with AES-256 before being persisted to document storage, and a ClinicalDocument record is created with `processingStatus = Pending`.

3. **Given** files are uploaded, **When** I view my dashboard, **Then** the upload history section shows each uploaded document with its file name, upload date, file size, and processing status (Pending / Processing / Completed / Failed).

4. **Given** a file exceeds the 25 MB limit or is not a PDF, **When** it is submitted, **Then** a clear per-file error message is displayed ("File too large" / "Only PDF files are accepted") and valid files in the same batch are uploaded successfully.

## Edge Cases

- What happens if upload fails halfway through a batch? (Already-uploaded files are stored; failed files are identified individually and can be retried without re-uploading the successful ones)
- What happens if document storage is temporarily unavailable? (Upload is rejected with a 503 response; user is shown a "Please try again shortly" message; no partial records are created)
