# User Story - US_046

## Story ID

- ID Format: US_046

## Story Title

- Role Assignment with Re-Authentication Gate

## Description

- As an authenticated Admin, I want to assign or modify the role (Staff or Admin) for any managed account — with mandatory re-authentication before role elevation to Admin or account deactivation — so that unauthorized privilege escalation is structurally prevented.

## Acceptance Criteria

1. **Given** I choose to change a user's role, **When** I select the new role and submit, **Then** the User record is updated with the new role; the change takes effect on the user's next session (existing JWT remains valid until expiry).

2. **Given** I attempt to elevate a user to the Admin role, **When** the action is initiated, **Then** a re-authentication modal prompts me to confirm my current password before the role change is committed.

3. **Given** I provide an incorrect password during re-authentication, **When** the verification fails, **Then** the role change is not committed, a failed re-auth event is logged in the audit trail, and the modal shows an error message.

4. **Given** a successful role change is committed, **When** the action completes, **Then** an audit log entry is created with `before state = previous role`, `after state = new role`, my Admin ID, and UTC timestamp.

## Edge Cases

- What happens when the re-authentication modal times out after 5 minutes without input? (The modal closes, the action is cancelled, and no role change is committed)
- What happens if the user being role-elevated is currently logged in? (The role change takes effect on their next session; their current session retains the old role until natural expiry or forced logout)
