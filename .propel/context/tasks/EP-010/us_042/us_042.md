# User Story - US_042

## Story ID

- ID: US_042

## Story Title

- Data Encryption at Rest and in Transit

## Description

- As a platform operator, I want all patient data encrypted at rest using AES-256 and in transit using TLS 1.2+, so that the platform meets HIPAA encryption requirements and protects patient data from unauthorized access.

## Acceptance Criteria

1. **Given** patient data is stored in PostgreSQL, **When** data is written to disk, **Then** it is encrypted at rest using AES-256 or equivalent encryption via PostgreSQL transparent data encryption or pgcrypto for sensitive fields.

2. **Given** data is transmitted between client and server, **When** any API request is made, **Then** the connection uses TLS 1.2 or higher, verified by inspecting response headers and certificate configuration.

3. **Given** clinical documents are stored, **When** a PDF is saved to storage, **Then** the file is encrypted at rest and the storage path references the encrypted object.

4. **Given** data is transmitted between microservices, **When** inter-service communication occurs, **Then** TLS encryption is enforced on all internal connections.

5. **Given** the encryption configuration is deployed, **When** a security audit reviews the configuration, **Then** no patient data is found in plaintext at rest or in transit.

## Edge Cases

- What happens when TLS certificate expires? (Automated renewal via Let's Encrypt or platform-managed certificates; health check monitors certificate expiry 30 days in advance)
- How does the system handle encryption key rotation? (Key management supports rotation without data re-encryption; new writes use new keys; reads support previous key versions)
