# User Story - US_043

## Story ID

- ID: US_043

## Story Title

- Immutable Audit Log Implementation

## Description

- As a compliance officer, I want an immutable append-only audit log recording all user actions on patient data with user ID, timestamp, action type, and affected record, so that the platform maintains a tamper-proof audit trail for HIPAA compliance.

## Acceptance Criteria

1. **Given** any user performs a create, read, update, or delete action on patient data, **When** the action occurs, **Then** the system creates an AuditLog entry with userId, action type, entityType, entityId, details (JSONB), IP address, and timestamp.

2. **Given** the AuditLog table is configured, **When** an UPDATE or DELETE operation is attempted on any audit log record, **Then** the database rejects the operation via INSERT-only enforcement (database trigger or policy).

3. **Given** audit logs are created, **When** the application writes via the repository, **Then** only INSERT operations are available (no update or delete methods exposed in the write-only repository pattern).

4. **Given** audit logs are stored, **When** retention policy is evaluated, **Then** records are retained for a minimum of 7 years in compliance with HIPAA requirements.

5. **Given** the audit log is queried, **When** a compliance review is performed, **Then** logs can be filtered by userId, action type, entityType, and date range with pagination support.

## Edge Cases

- What happens when the audit log insert fails? (Operation is retried with exponential backoff; if persistent failure, the original user action still succeeds but alerts are generated for the missing audit entry)
- How does the system handle audit log storage growth over 7 years? (Archival strategy with partitioned tables by year; older partitions can be moved to cold storage while maintaining query access)
