# User Story - US_045

## Story ID

- ID: US_045

## Story Title

- Input Validation and API Rate Limiting

## Description

- As a platform operator, I want input validation and output encoding on all user-facing inputs to prevent injection attacks, plus rate limiting on public-facing API endpoints to prevent abuse, so that the platform is protected against OWASP A03 injection and brute-force attacks.

## Acceptance Criteria

1. **Given** any user-facing API endpoint, **When** a request is received, **Then** all input fields are validated using FluentValidation with type checking, length limits, and format enforcement before processing.

2. **Given** user input is processed, **When** the response is generated, **Then** output encoding is applied to prevent XSS and injection in all rendered content.

3. **Given** a malicious input is submitted (SQL injection, XSS payload), **When** validation runs, **Then** the system rejects the request with a 400 Bad Request response and logs the attempt without exposing internal details.

4. **Given** public-facing API endpoints exist (login, registration), **When** a client exceeds the configured rate limit, **Then** the system returns 429 Too Many Requests with a Retry-After header.

5. **Given** rate limiting is configured, **When** different endpoints are evaluated, **Then** authentication endpoints have stricter limits (e.g., 10 requests/minute) than general read endpoints (e.g., 100 requests/minute).

## Edge Cases

- What happens when a legitimate user is rate-limited during peak usage? (Rate limits are per-IP or per-user; authenticated users get higher limits; 429 response includes retry guidance)
- How does the system handle encoded or obfuscated injection attempts? (Multiple decoding passes before validation; defense-in-depth via parameterized queries at the database layer)
