# User Story - US_046

## Story ID

- ID: US_046

## Story Title

- AI Security Controls (PII and Access)

## Description

- As a compliance officer, I want PII redacted from AI prompts sent to non-BAA providers and document-level access control enforced in AI retrieval pipelines, so that patient data is protected during AI processing per HIPAA requirements.

## Acceptance Criteria

1. **Given** an AI prompt is constructed from patient data, **When** the prompt is sent to an external AI provider without a HIPAA BAA, **Then** all PII (names, dates of birth, SSNs, contact information) is redacted or de-identified before transmission.

2. **Given** the prompt is sent to Azure OpenAI with a HIPAA BAA, **When** the prompt includes patient data, **Then** PII may be included as it is covered by the BAA agreement.

3. **Given** the AI retrieval pipeline processes a query, **When** document chunks are retrieved, **Then** the system filters results to only include documents the requesting user is authorized to access based on their role and patient scope.

4. **Given** a Staff user queries the AI for patient data, **When** the retrieval pipeline runs, **Then** only documents for the specific patient being reviewed are included in the AI context (no cross-patient data leakage).

5. **Given** PII redaction is applied, **When** the AI response is received, **Then** the system re-associates the redacted fields with original values for display without exposing PII in AI provider logs.

## Edge Cases

- What happens when PII redaction misses an embedded PII reference in unstructured text? (Defense-in-depth: BAA path is preferred for production; redaction uses NER-based detection with regex fallback)
- How does the system handle AI responses that hallucinate PII not present in source documents? (Content filtering flags AI outputs containing potential PII not from source; flagged for staff review)
