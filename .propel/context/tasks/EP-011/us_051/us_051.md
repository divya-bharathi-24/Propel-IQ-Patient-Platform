# User Story - US_051

## Story ID

- ID Format: US_051

## Story Title

- API Performance Instrumentation & Structured Logging

## Description

- As a platform engineer, I want all API endpoints instrumented for p95 latency tracking (≤2s target), Serilog structured logging with correlation ID propagation, MediatR CQRS separation, and FluentValidation on all endpoints, so that performance SLAs are measurable and requests are fully traceable across the system.

## Acceptance Criteria

1. **Given** any API endpoint receives a request, **When** the response is returned, **Then** a structured Serilog log entry is written containing: correlation ID, route, HTTP method, status code, duration (ms), user ID, and role.

2. **Given** a correlation ID is injected at the API Gateway, **When** a request traverses through module handlers, **Then** the same correlation ID appears in all log entries related to that request — including database calls and external service calls.

3. **Given** MediatR CQRS is configured, **When** I inspect module implementations, **Then** all write operations use Commands and all read operations use Queries — with no direct repository calls from controllers.

4. **Given** any endpoint receives invalid input, **When** FluentValidation runs, **Then** a structured HTTP 400 response is returned with per-field validation errors and the validation failure is logged with the correlation ID.

## Edge Cases

- What happens when a request exceeds the 2-second SLA? (An automated alert is triggered when p95 latency breaches 2s over a 5-minute window; the alert is sent to the configured monitoring channel)
- What happens when correlation ID is missing from an incoming request? (The gateway generates a new correlation ID and attaches it before routing; the generated ID is returned in the response header `X-Correlation-Id`)
