# User Story - US_053

## Story ID

- ID Format: US_053

## Story Title

- Frontend Performance & Horizontal Scalability Baseline

## Description

- As a platform engineer, I want the Angular SPA to meet Core Web Vitals targets (LCP <2.5s, FID <100ms, CLS <0.1), the backend to support 100+ concurrent users with horizontal scaling readiness, and PDF processing to handle up to 50 MB without timeout, so that the platform reliably handles production load from day one.

## Acceptance Criteria

1. **Given** the Angular SPA is deployed to Netlify/Vercel CDN, **When** measured with Lighthouse or Core Web Vitals tooling, **Then** the initial page load scores LCP <2.5s, FID <100ms, and CLS <0.1 on a standard broadband connection.

2. **Given** a load test simulates 100 concurrent users performing booking and slot queries simultaneously, **When** the test runs for 5 minutes, **Then** p95 API response time remains ≤2s for non-AI endpoints and no HTTP 5xx errors occur.

3. **Given** the backend is stateless (no in-process session state), **When** the backend is horizontally scaled to two instances, **Then** both instances serve requests correctly using shared Redis session state and PostgreSQL — with no request failing due to session affinity requirements.

4. **Given** a PDF document of up to 50 MB is submitted for upload and processing, **When** the extraction pipeline runs, **Then** it completes without HTTP timeout, memory exhaustion, or OOM error — processing time is logged for SLA monitoring.

## Edge Cases

- What happens when Core Web Vitals degrade after a new frontend deployment? (Lighthouse CI check in the GitHub Actions pipeline fails the deployment if Core Web Vitals breach targets; the previous deployment remains live)
- What happens when a 50 MB PDF causes processing latency beyond the AI p95 target of 30 seconds? (AI p95 latency is tracked separately from document upload; progress indicator is displayed for operations >5s per NFR-002; SLA breach is logged)
