# User Story - US_006

## Story ID

- ID Format: US_006

## Story Title

- Core Patient, User, Appointment & Specialty Entity Models

## Description

- As a developer, I want EF Core 9 entity classes and fluent configuration for Patient, User, Appointment, WaitlistEntry, and Specialty domain entities, so that all authentication and booking features have a reliable, integrity-enforced data foundation.

## Acceptance Criteria

1. **Given** the EF Core 9 context is configured, **When** I run `dotnet ef migrations add Initial`, **Then** a migration script is generated that creates Patient, User, Appointment, WaitlistEntry, and Specialty tables with all columns, constraints, and foreign keys correctly defined.

2. **Given** the Patient and Appointment entities are configured, **When** a Patient record is soft-deleted, **Then** the `status` field is set to `Deactivated` and the row is retained; no hard DELETE is issued.

3. **Given** the Appointment entity is configured with optimistic concurrency, **When** two concurrent requests attempt to book the same slot, **Then** only one succeeds; the other receives a `DbUpdateConcurrencyException`.

4. **Given** the data model is applied, **When** I query related entities, **Then** FK constraints are enforced (e.g., inserting an Appointment with a non-existent `patientId` raises a constraint violation).

## Edge Cases

- What happens when a migration is applied to a database that already has data? (Migrations are backwards-compatible; additive changes only; destructive changes go through separate explicit migration scripts)
- How is email uniqueness for Patient enforced at the DB level? (Unique index on `Patient.email` column; EF Core `HasIndex().IsUnique()` configuration)
