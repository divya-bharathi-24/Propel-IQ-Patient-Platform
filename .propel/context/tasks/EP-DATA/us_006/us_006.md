# User Story - US_006

## Story ID

- ID: US_006

## Story Title

- Patient and User Entity Models

## Description

- As a developer, I want the Patient and User domain entities implemented with EF Core code-first configuration including email uniqueness constraints, soft deletion support, and relationship mappings, so that user identity and patient profile data can be persisted with referential integrity.

## Acceptance Criteria

1. **Given** the EF Core DbContext is configured, **When** the Patient entity is defined, **Then** it includes all attributes (id, name, email, phone, dateOfBirth, passwordHash, emailVerified, status, createdAt) with email configured as a unique index.

2. **Given** the Patient entity is defined, **When** a duplicate email insert is attempted, **Then** the database rejects the operation with a unique constraint violation.

3. **Given** the User entity is defined, **When** it is configured, **Then** it includes all attributes (id, email, passwordHash, role, status, lastLoginAt, createdAt) with email as a unique index and role restricted to Patient/Staff/Admin values.

4. **Given** soft deletion is implemented, **When** a Patient record is deleted via the application, **Then** the record is flagged as Deactivated (not physically removed) and excluded from default queries via a global query filter.

5. **Given** the entities are configured, **When** a migration is generated and applied, **Then** the Patient and User tables are created with correct column types, constraints, and indexes.

## Edge Cases

- What happens when a Patient record references a non-existent User? (Foreign key constraint prevents orphaned references; migration enforces FK relationship)
- How does the system handle concurrent updates to the same Patient record? (Optimistic concurrency via EF Core concurrency token; second update receives conflict exception)
