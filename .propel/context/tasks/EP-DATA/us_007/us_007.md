# User Story - US_007

## Story ID

- ID Format: US_007

## Story Title

- Clinical, AI & Queue Entity Models

## Description

- As a developer, I want EF Core entity classes for IntakeRecord, ClinicalDocument, ExtractedData, DataConflict, MedicalCode, NoShowRisk, and QueueEntry, so that clinical intelligence and queue management features have a persisted, AI-compatible data layer.

## Acceptance Criteria

1. **Given** the entities are configured, **When** I run migrations, **Then** IntakeRecord JSONB columns (`demographics`, `medicalHistory`, `symptoms`, `medications`) are mapped as JSONB in PostgreSQL and are queryable via EF Core.

2. **Given** the ExtractedData entity is configured, **When** I store a pgvector embedding, **Then** the `embedding` vector column accepts float arrays of the configured dimension and the `pgvector` extension is enabled in the migration.

3. **Given** MedicalCode entity is persisted, **When** I insert a code with `verificationStatus = Pending`, **Then** the record is retrievable with all required fields (codeType, code, description, confidence, sourceDocumentId, verifiedBy, verifiedAt).

4. **Given** all clinical entities are configured, **When** I verify FK relationships, **Then** every ExtractedData row references a valid ClinicalDocument and Patient; orphan records are prevented by FK constraints.

## Edge Cases

- What happens when a confidence score outside 0–1 range is stored? (Check constraint on `confidence` column enforces 0 ≤ value ≤ 1 at the database level)
- How is the JSONB field validated at the application layer? (FluentValidation schemas enforce required sub-fields before persisting)
