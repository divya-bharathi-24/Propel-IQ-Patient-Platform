# User Story - US_008

## Story ID

- ID: US_008

## Story Title

- Intake and Insurance Data Entities

## Description

- As a developer, I want the IntakeRecord, InsuranceValidation, and CalendarSync entities implemented with JSONB fields and relationship mappings, so that patient intake data, insurance validation results, and calendar sync records can be persisted with flexible schema support.

## Acceptance Criteria

1. **Given** the IntakeRecord entity is defined, **When** it is configured, **Then** it includes JSONB columns for demographics, medicalHistory, symptoms, and medications with a source indicator (AI/Manual) and FK relationships to Patient and Appointment.

2. **Given** the InsuranceValidation entity is defined, **When** it is configured, **Then** it includes providerName, insuranceId, result (Matched/NotMatched), and validatedAt with a many-to-one relationship to Patient.

3. **Given** the CalendarSync entity is defined, **When** it is configured, **Then** it includes provider (Google/Outlook), externalEventId, syncStatus (Synced/Failed/Revoked), and syncedAt with FK relationships to Patient and Appointment.

4. **Given** JSONB fields are configured, **When** intake data is stored with nested JSON objects, **Then** the data is persisted correctly and queryable via EF Core JSONB operators.

5. **Given** all entities are configured, **When** a migration is generated and applied, **Then** all tables are created with correct column types, FK constraints, and indexes on patientId columns.

## Edge Cases

- What happens when JSONB intake data contains unexpected field names? (JSONB is schema-flexible; application-level validation via FluentValidation ensures required fields; extra fields are preserved)
- How does the system handle a CalendarSync record for a deleted Appointment? (Soft deletion preserves Appointment; CalendarSync retains reference for audit trail)
