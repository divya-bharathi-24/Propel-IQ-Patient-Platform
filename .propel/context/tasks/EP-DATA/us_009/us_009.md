# User Story - US_009

## Story ID

- ID Format: US_009

## Story Title

- pgvector Extension, Seed Data & Database Encryption

## Description

- As a developer, I want the pgvector extension enabled in the database migration, pgcrypto configured for AES-256 at-rest encryption, and seed data for Specialty and dummy InsuranceProvider records, so that AI vector search and insurance pre-check features work immediately in all environments.

## Acceptance Criteria

1. **Given** migrations are applied, **When** I check the PostgreSQL extensions, **Then** both `pgvector` and `pgcrypto` extensions are listed as active in the target database.

2. **Given** seed data migrations are applied, **When** I query the Specialty reference table, **Then** at least 5 predefined medical specialties are present (e.g., General Practice, Cardiology, Dermatology, Orthopedics, Pediatrics).

3. **Given** the seed migration for insurance records is applied, **When** I query the internal insurer dataset, **Then** at least 10 dummy insurer records are present with name and insurer ID fields for pre-check matching.

4. **Given** AES-256 at-rest encryption is configured via pgcrypto, **When** sensitive patient columns are stored, **Then** the encryption function is applied at the application layer before DB write, and decryption restores the original value correctly.

## Edge Cases

- What happens if pgvector is not available on the database host? (Migration detects extension availability; logs a startup warning; AI features degrade gracefully with a manual fallback flag)
- How is idempotency of seed data ensured across re-runs? (Seed migrations use INSERT IF NOT EXISTS or UPSERT patterns to prevent duplicate records)
