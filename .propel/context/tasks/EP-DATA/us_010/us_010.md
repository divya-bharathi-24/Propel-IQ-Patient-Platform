# User Story - US_010

## Story ID

- ID: US_010

## Story Title

- Database Migration Pipeline and Seed Data

## Description

- As a developer, I want versioned EF Core migration scripts with zero-downtime support and predefined seed data including dummy insurance records and specialties, so that the database can be reliably evolved and development/testing has consistent reference data.

## Acceptance Criteria

1. **Given** all domain entities are defined, **When** an initial migration is generated, **Then** the migration creates all tables with correct schemas, indexes, constraints, and relationships in a single transactional migration.

2. **Given** the migration pipeline is configured, **When** a schema change is made to an entity, **Then** a new versioned migration is generated that can be applied without dropping existing data (zero-downtime compatible).

3. **Given** seed data is configured, **When** the database is initialized, **Then** predefined dummy insurance records (minimum 10 entries with provider names and insurance IDs) are inserted for validation testing.

4. **Given** seed data is configured, **When** the database is initialized, **Then** predefined Specialty records (e.g., General Practice, Cardiology, Dermatology, Orthopedics, Pediatrics) are inserted for appointment booking.

5. **Given** the migration history is tracked, **When** `dotnet ef migrations list` is executed, **Then** all applied and pending migrations are listed with timestamps showing the migration order.

## Edge Cases

- What happens when a migration conflicts with existing data? (Migration includes data transformation steps; rollback migration is generated for reversibility)
- How does the system handle seed data re-execution on an already seeded database? (Seed operations use `HasData` or upsert patterns to prevent duplicate inserts)
