# User Story - US_002

## Story ID

- ID: US_002

## Story Title

- .NET 8 Backend Solution Scaffold

## Description

- As a developer, I want a properly structured .NET 8 ASP.NET Core Web API solution with microservice project structure, MediatR CQRS pipeline, and FluentValidation integration, so that backend development follows clean architecture patterns with consistent command/query separation.

## Acceptance Criteria

1. **Given** the project repository is cloned, **When** the .NET 8 solution is created, **Then** it contains separate projects for Auth, Patient, Appointment, Clinical, AI, Notification, and Admin services with a shared kernel/common project.

2. **Given** the solution is created, **When** MediatR is configured, **Then** each service project has a CQRS pipeline with sample command and query handlers demonstrating the pattern.

3. **Given** the solution is created, **When** FluentValidation is configured, **Then** request validation is integrated into the MediatR pipeline as a behavior, automatically validating all incoming commands.

4. **Given** the solution is built, **When** `dotnet build` is executed, **Then** all projects compile without errors and the solution structure follows the .NET project naming conventions.

5. **Given** the API gateway project is configured, **When** it is started, **Then** it routes requests to downstream service projects and returns a health check response at `/health`.

## Edge Cases

- What happens when a service project has a missing dependency reference? (Solution-level `Directory.Build.props` manages shared package versions to prevent version drift)
- How does the system handle conflicting NuGet package versions across services? (Central package management via `Directory.Packages.props` enforces version consistency)
