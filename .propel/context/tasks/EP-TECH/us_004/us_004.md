# User Story - US_004

## Story ID

- ID Format: US_004

## Story Title

- GitHub Actions CI/CD Pipeline

## Description

- As a development team, I want a GitHub Actions CI/CD pipeline that runs lint, build, and test gates on every pull request and deploys automatically to staging on merge, so that code quality is enforced consistently and releases are reliable.

## Acceptance Criteria

1. **Given** a pull request is opened against the main branch, **When** the pipeline triggers, **Then** it runs Angular lint, .NET build, xUnit unit tests, and Playwright E2E smoke tests — failing the PR if any gate fails.

2. **Given** all gates pass on a merged PR, **When** the deploy job runs, **Then** the Angular SPA deploys to Netlify/Vercel and the .NET API Docker image deploys to Railway within 10 minutes of merge.

3. **Given** the pipeline is configured, **When** I inspect the workflow YAML, **Then** secrets (API keys, deploy tokens) are stored as GitHub repository secrets and never appear in plain text in logs.

4. **Given** the E2E test gate runs, **When** Playwright tests execute, **Then** all scaffold smoke tests pass and a test report artifact is uploaded to the workflow run.

## Edge Cases

- What happens if the Railway deployment fails mid-deploy? (Pipeline marks the deploy step as failed; previous deployment remains live; notification logged to the workflow run summary)
- How are database migration failures handled in CI? (A dedicated migration smoke test runs after deploy; failure triggers an alert and blocks traffic cutover)
