# User Story - US_005

## Story ID

- ID Format: US_005

## Story Title

- Netlify/Railway Deployment Configuration

## Description

- As a developer, I want Netlify (frontend) and Railway (backend) deployment targets configured for the platform, so that the team has a live staging environment matching production constraints from the beginning of development.

## Acceptance Criteria

1. **Given** the Netlify site is configured, **When** a build artifact is pushed, **Then** the Angular SPA is served via global CDN with a valid HTTPS certificate and LCP < 2.5s for the shell page.

2. **Given** the Railway project is configured, **When** the .NET 9 Docker container is deployed, **Then** the API is reachable at the configured domain over TLS 1.2+ and returns HTTP 200 on the `/health` endpoint.

3. **Given** environment variables are configured in both platforms, **When** the application starts, **Then** it reads all required configuration values (DB connection, Redis URL, JWT secrets) from the environment without hardcoded fallbacks.

4. **Given** the deployment is live, **When** I access the API from the Netlify-hosted frontend, **Then** CORS headers are correctly configured to allow requests only from the Netlify domain.

## Edge Cases

- What happens if Railway free-tier limits are reached? (Documented migration path to paid tier; monitoring alert set at 80% resource usage)
- How are CORS misconfigurations caught? (E2E smoke test from the deployed frontend verifies a cross-origin API call succeeds)
