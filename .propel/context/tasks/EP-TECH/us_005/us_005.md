# User Story - US_005

## Story ID

- ID: US_005

## Story Title

- CI/CD, Logging and Test Framework Setup

## Description

- As a developer, I want GitHub Actions CI/CD pipelines, Serilog structured logging with correlation IDs, and xUnit/Playwright test frameworks configured, so that the team has automated build/test pipelines, observable logging, and testing infrastructure from day one.

## Acceptance Criteria

1. **Given** the repository is configured, **When** a pull request is created, **Then** GitHub Actions triggers a CI pipeline that builds both Angular frontend and .NET backend, runs unit tests, and reports pass/fail status.

2. **Given** the CI pipeline is configured, **When** tests fail, **Then** the pipeline fails with clear error output identifying the failing test and the PR is blocked from merge.

3. **Given** Serilog is configured, **When** an API request is processed, **Then** structured log entries include a correlation ID, timestamp, log level, and request context propagated across all microservice calls.

4. **Given** the xUnit test framework is set up, **When** a developer creates a test class in any service project, **Then** tests are discoverable and executable via `dotnet test` with code coverage reporting.

5. **Given** Playwright is configured, **When** a developer creates a `.spec.ts` file, **Then** the E2E test is executable against the Angular application with browser automation and screenshot capture on failure.

## Edge Cases

- What happens when GitHub Actions runner quota is exhausted? (Pipeline queues; team is notified via GitHub status check; manual fallback build documented)
- How does the system handle Serilog sink failures (e.g., log storage unavailable)? (Fallback to console sink; application continues without blocking on log delivery)
